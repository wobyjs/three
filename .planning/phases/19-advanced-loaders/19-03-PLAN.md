# Phase 19 Plan 03: Advanced Texture-Format Loader Batch

## Context

Final plan of Phase 19. 19-01 covers 3D-model-format loaders, 19-02 covers animation/skinning loaders. This plan rounds out the phase with 4 advanced texture-decoder demos that all use a `*Loader` from `examples/jsm/loaders/` (or sibling packages) to decode a non-trivial image format into a `Texture` and apply it to a sphere or plane for visualization:

- **KTX** (`KTXLoader` — Khronos Texture container, hardware-compressed BC/ETC/ASTC inside)
- **PVRTC** (`PVRLoader` — PowerVR mobile-GPU compressed texture)
- **TIFF** (`TIFFLoader` — multi-page TIFF decoded via `utif` or `tiff.js`)
- **UltraHDR** (`UltraHDRLoader` — gainmap-encoded HDR JPEG, sister to the `HDRJPGLoader` shipped in Phase 18 Plan 01)

All 4 IDs were **verified absent from `demo/src/registry.ts`** at planning time (2026-06-26 grep, see 19-CONTEXT.md). All 4 IDs **confirmed present in upstream `files.json`**. All 4 have **pre-converted scaffolds** under `code/examples/webgl/loaders/` — scaffolds are stub-quality (no actual loader call, often a placeholder white sphere); implementer MUST consult upstream HTML.

## Locked Decisions (from Phase 15 RESEARCH / Phase 17 + 18 SUMMARYs)

- **init3D pattern**: `const init3D = (container: HTMLElement) => { ... }` with module-level `let _cleanupFn`. JSX wraps with `<div ref={(el) => { if (el) init3D(el) }} />`.
- **JSX pragma**: every file begins with `/** @jsxImportSource woby */`.
- **Forbidden**: `as any`, `useEffect`, `useRef` for plain objects.
- **Visual verification**: dv profile-4 screenshot → Claude approval → THEN register.
- **Dev server**: http://localhost:5300, profile-4 only.
- **Asset paths**: `https://threejs.org/examples/textures/...`.
- **Compressed-texture caveat (Tasks 1 + 2)**: KTX and PVRTC require GPU support for the contained format. KTX containers typically wrap ETC2/ASTC/BC formats; PVRTC is PowerVR-specific (iOS-only). On a desktop Chrome on profile-4 with a typical discrete GPU, BC/ASTC variants will likely work; PVRTC will likely fail to upload (no decoder). Document the unsupported-format fallback path in SUMMARY for any task whose format isn't supported by the executor's GPU.

### Task 1: Port WebGLLoaderTextureKTX

**File**: `demo/src/WebGLLoaderTextureKTX.tsx`
**Source**: threejs.org/examples/#webgl_loader_texture_ktx
**Reference scaffold**: `code/examples/webgl/loaders/LoaderTextureKTX.tsx` (stub only)

**Workflow**:
1. Open upstream HTML at `https://threejs.org/examples/#webgl_loader_texture_ktx` and view-source for: `KTXLoader` import (`three/examples/jsm/loaders/KTXLoader.js`), the demo `.ktx` URL(s) (often multiple KTX files demonstrating different internal formats like `disturb_BC1.ktx`, `disturb_BC3.ktx`, `disturb_ETC2.ktx`, `disturb_ASTC.ktx` at `https://threejs.org/examples/textures/compressed/`), the multi-sphere layout (one sphere per format).
2. Read the scaffold to see existing imports.
3. Write `demo/src/WebGLLoaderTextureKTX.tsx`:
   - `/** @jsxImportSource woby */` header
   - Module-level `let _cleanupFn: (() => void) | undefined`
   - `init3D(container)`:
     - Scene + PerspectiveCamera + WebGLRenderer (append to container)
     - `const ktxLoader = new KTXLoader()`
     - Detect GPU capabilities via `renderer.extensions.has('WEBGL_compressed_texture_s3tc')`, `.has('WEBGL_compressed_texture_etc')`, `.has('WEBGL_compressed_texture_astc')` to pick which KTX variants to try
     - For each supported variant, `ktxLoader.load(url, (texture) => { texture.minFilter = LinearMipmapLinearFilter; mesh.material.map = texture; mesh.material.needsUpdate = true; })`
     - Layout: one `SphereGeometry(1, 32, 32)` per variant arranged in a row at `x = -3, 0, 3, ...` so the user can compare visual quality
     - Lights: AmbientLight + DirectionalLight (so the textured spheres are lit)
     - `OrbitControls`, framed on the row of spheres
     - Cleanup: dispose all KTX textures + geometries + materials + controls + canvas
   - JSX root with container ref-callback
4. Typecheck. No `as any`.
5. dv visual verify (recipe below). `<id>` = `webgl_loader_texture_ktx`. KTX fetches are small + fast; wait 2s after navigate.
6. Claude reviews. PASS criteria: at least ONE sphere with a recognizable texture (likely the "disturb" noise pattern) is visible. If multiple supported variants are loaded, they should look visually similar. If ALL formats are unsupported by the executor's GPU, document the unsupported-format failure mode and pass with a console-error-only screenshot — note in SUMMARY.
7. Register: `{ id: 'webgl_loader_texture_ktx', name: 'KTX Compressed Texture', category: 'loaders', component: () => import('./WebGLLoaderTextureKTX') }`.

## Risk Notes

- **KTX**: Multi-variant KTX file set on threejs.org is reliable. GPU-format support detection is important — without it, you get cryptic WebGL errors. Use `renderer.extensions.has(...)` checks.
- **PVRTC**: Effectively guaranteed to NOT decode on the executor's Windows desktop Chrome — the test for this demo is the EXPLANATORY FALLBACK overlay and the absence of unhandled exceptions, NOT actual textured spheres. Set expectations accordingly. The PVRLoader parse path still runs.
- **TIFF**: `utif` peer dep must be installed if not already (verify on npmjs first). TIFF decode is single-threaded JS — large multi-page TIFFs can stall; pick small upstream samples.
- **UltraHDR**: This is a NEW-ish loader (introduced in three r161+); pin the installed three version BEFORE assuming the loader path. If `three/examples/jsm/loaders/UltraHDRLoader.js` doesn't exist in the installed three, EITHER upgrade three (verify no breaking changes across other demos first), OR fall back to the HDRJPGLoader from Phase 18 Plan 01 and document the substitution. The pedagogical goal (gainmap-encoded HDR JPG decode → PMREM env → reflective sphere) is preserved by either loader.
