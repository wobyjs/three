# Phase 19 Plan 01: 3D Model Format Loaders Batch

## Context

First plan of Phase 19. Phase 18 closed at PASS 7/7 with 3 plans of 4 demos each (12 material/texture demos). Phase 19 mirrors that cadence with 3 plans × 4 demos = 12 loader demos. This plan groups 4 large-asset 3D-model-format loaders that all share the "load asset → add to scene → orbit/inspect" lifecycle pattern but exercise distinct loader classes from `three/examples/jsm/loaders/` (or sibling packages):

- **3DTiles** (`3DTilesRenderer` from `3d-tiles-renderer`): streamed Cesium tilesets with LOD swapping
- **IFC** (`IFCLoader` from `web-ifc-three`): industry-standard BIM/architecture format
- **LDraw** (`LDrawLoader` from `three/examples/jsm/loaders/LDrawLoader.js`): LEGO part assembly with part library
- **USDZ** (`USDZLoader` from `three/examples/jsm/loaders/USDZLoader.js`): Apple AR Quick Look format

All 4 IDs were **verified absent from `demo/src/registry.ts`** at planning time (2026-06-26 grep, see 19-CONTEXT.md "Pre-Flight Scope Verification"). All 4 IDs **confirmed present in upstream `threejs.org/examples/files.json`** at planning time. All 4 have **pre-converted `.tsx` scaffolds** under `code/examples/webgl/loaders/` (table in 19-CONTEXT.md) — but per Phase-15/17/18 precedent, those scaffolds are stub-quality only (often the loader is not even called, asset paths are wrong, or it renders a placeholder mesh). The implementer MUST consult upstream `threejs.org/examples/#<id>` HTML for canonical behavior.

## Locked Decisions (from Phase 15 RESEARCH / Phase 17 + 18 SUMMARYs)

- **init3D pattern**: `const init3D = (container: HTMLElement) => { ... }` with module-level `let _cleanupFn`. JSX wraps with `<div ref={(el) => { if (el) init3D(el) }} />`.
- **JSX pragma**: every file begins with `/** @jsxImportSource woby */` (NOT `@woby/three`).
- **Forbidden**: `as any`, `useEffect`, `useRef` for plain objects.
- **Visual verification**: dv profile-4 screenshot → Claude approval → THEN register in `demo/src/registry.ts`.
- **Dev server**: http://localhost:5300, profile-4 only (user constraint → sequential execution even though planning marks wave: 1).
- **Asset paths**: `https://threejs.org/examples/models/...` (large model URLs). If a model is too large for fast iteration OR returns 404, substitute with a smaller placeholder from the same loader family AND document the substitution in SUMMARY.
- **No editing of upstream `code/examples/`** — those are reference. New files go in `demo/src/`.

## Risk Notes

- **3DTiles**: Google Photorealistic 3D Tiles require a billable API key — do NOT use those. Stick to free public Cesium ION sample tilesets or a self-hosted small tileset. Network bandwidth can be heavy; if the tileset never finishes the first LOD load, substitute a smaller tileset.
- **IFC**: WASM loader path is fragile. If `setWasmPath('https://unpkg.com/web-ifc@<version>/')` 404s, pin to the version recorded in `package-lock.json` after install. Some IFC files are huge (100MB+) — start with the smallest public sample.
- **LDraw**: The official parts-library mirror occasionally rate-limits. The `MPDLoader` triggers dozens of parallel fetches; if many 404 in console, the model will render with missing colored bricks — that may still be acceptable visual proof of the loader working, but document in SUMMARY.
- **USDZ**: USDZ files are zip-packaged USD archives. The Three.js `USDZLoader` supports a narrow USDZ subset (no animations, limited materials). The shrimp model is the canonical safe choice.
