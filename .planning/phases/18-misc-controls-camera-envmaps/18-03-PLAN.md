## Tasks

### Task 1: Port WebGLMaterialsTextureManualMipmap

**File**: `demo/src/WebGLMaterialsTextureManualMipmap.tsx`
**Source**: threejs.org/examples/#webgl_materials_texture_manualmipmap
**Reference scaffold**: `code/examples/webgl/materials/MaterialsTextureManualmipmap.tsx` (stub creates one solid canvas, no mipmap chain)
**Key features**:
- Create a base canvas (e.g. 256×256) AND each successive mip level canvas (128×128, 64×64, 32×32, 16×16, 8×8, 4×4, 2×2, 1×1) where EACH LEVEL is painted with a visually distinct color/pattern (upstream typically uses different solid colors per mip — red 256, green 128, blue 64, etc.) so when the camera zooms in/out you visibly see different mip levels selected
- Build a `THREE.DataTexture` or `THREE.CanvasTexture`, then manually populate `texture.mipmaps = [canvas0, canvas1, canvas2, ...]`
- Set `texture.minFilter = THREE.NearestMipmapNearestFilter` (or `LinearMipmapLinearFilter`) so mipmap selection is active
- Set `texture.generateMipmaps = false` — critical, otherwise WebGL overwrites our manually-authored chain with auto-generated downsamples
- Apply to a large `PlaneGeometry` (e.g. 20×20) viewed at a glancing angle so distance varies across the surface
- Camera positioned so the user can see multiple mip levels active at once (foreground = base, midground = mip 2, far = mip 5)
- `OrbitControls` to let user pan and watch mip swaps
- Cleanup: dispose texture, geometry, material, controls, canvas
**Registry ID**: `webgl_materials_texture_manualmipmap`
**Registry name**: `'Manual Mipmap Chain (Per-Level Colors)'`
**Category**: `materials`

### Task 2: Port WebGLMaterialsTexturePartialUpdate

**File**: `demo/src/WebGLMaterialsTexturePartialUpdate.tsx`
**Source**: threejs.org/examples/#webgl_materials_texture_partialupdate
**Reference scaffold**: `code/examples/webgl/materials/MaterialsTexturePartialupdate.tsx` (stub renders a solid-color plane, no partial-update logic)
**Key features**:
- Demonstrate the partial-upload optimization: instead of re-uploading the whole texture each frame, only push a small sub-rect (e.g. a 16×16 patch) to GPU per frame via `renderer.copyTextureToTexture(position, srcTexture, dstTexture, level)` OR `gl.texSubImage2D` under the hood
- Upstream pattern: a large destination `DataTexture` (e.g. 256×256) is created once and applied to a plane; a small source texture (e.g. a `CanvasTexture` containing a moving 16×16 dot) is generated per-frame; each frame `renderer.copyTextureToTexture(new THREE.Vector2(x, y), srcTexture, dstTexture)` blits the small patch into the destination at a moving `(x, y)` offset — visually you see a trail/painting effect accumulating on the surface
- Initialize the destination texture with a known background (e.g. dark blue) so the painted-on patches are visually distinct
- Position the camera so the textured plane fills most of the viewport
- `OrbitControls`
- Animation loop: update the source texture content + offset position + call `copyTextureToTexture` each frame
- Cleanup: dispose source/destination textures, geometry, material, controls, canvas
**Registry ID**: `webgl_materials_texture_partialupdate`
**Registry name**: `'Partial Texture Update (copyTextureToTexture)'`
**Category**: `materials`

### Task 3: Port WebGLMaterialsVideoWebcam

**File**: `demo/src/WebGLMaterialsVideoWebcam.tsx`
**Source**: threejs.org/examples/#webgl_materials_video_webcam
**Reference scaffold**: `code/examples/webgl/materials/MaterialsVideoWebcam.tsx` (stub renders a dark plane, no getUserMedia)
**Key features**:
- Request webcam access via `navigator.mediaDevices.getUserMedia({ video: true, audio: false })` and attach the returned `MediaStream` to a hidden `<video>` element (`video.srcObject = stream; video.play()`)
- Wrap the video element with `new THREE.VideoTexture(video)` and apply to a `MeshBasicMaterial({ map: videoTexture })` on a plane or cube — VideoTexture handles per-frame updates automatically
- **Permission handling**: getUserMedia returns a Promise that REJECTS if the user denies the prompt. Wrap in try/catch and on rejection show a friendly DOM overlay ("Webcam access denied — please grant permission and reload") so the demo never silently breaks. Also handle the case where the user is on `http://` (insecure context) — getUserMedia requires `https://` OR `localhost`; since dev server is `http://localhost:5300` this is fine, but verify
- Apply the video texture to a flat plane (typical aspect ratio match: PlaneGeometry(4, 3) for a 4:3 webcam, or PlaneGeometry(16, 9) for a 16:9 webcam) OR to a `BoxGeometry` cube so the video appears on multiple faces simultaneously (upstream often does the cube variant for visual interest)
- `OrbitControls`
- Slight per-frame rotation of the mesh so the video surface is visible from multiple angles
- Cleanup: STOP the MediaStream tracks (`stream.getTracks().forEach(t => t.stop())`) to release the webcam hardware, remove the hidden video element, dispose VideoTexture, geometry, material, controls, canvas
- **Verification permission**: When dv profile-4 (Chrome) navigates to this demo, the browser will prompt for webcam access. The dv CLI cannot auto-approve permission prompts. The implementer should either:
  - (a) Pre-grant webcam permission on the profile-4 Chrome profile before navigating (set `chrome://settings/content/camera` to allow `http://localhost:5300`), OR
  - (b) Accept that the screenshot will capture the permission-denied fallback overlay rather than the actual webcam feed, and add a manual-verification step where the developer briefly grants permission via headed-mode Chrome and re-captures
  - Recommended: (a) — one-time profile setup, then full automation. Document the setup step in SUMMARY.
**Registry ID**: `webgl_materials_video_webcam`
**Registry name**: `'Webcam Video Texture'`
**Category**: `materials`

### Task 4: Port WebGLMaterialsNormalmapObjectSpace

**File**: `demo/src/WebGLMaterialsNormalmapObjectSpace.tsx`
**Source**: threejs.org/examples/#webgl_materials_normalmap_object_space
**Reference scaffold**: `code/examples/webgl/materials/MaterialsNormalmapObjectSpace.tsx` (stub uses a stock material — does NOT apply an object-space normal map)
**Key features**:
- Object-space normal mapping differs from tangent-space normal mapping in that the normal vectors are stored in object-local coordinates (not surface-tangent coordinates), encoded into RGB where R = x, G = y, B = z (typically with -1..1 mapped to 0..1 in 8-bit). This eliminates the need for tangent/bitangent attributes but only works for non-deforming meshes (no skinning, no morph targets)
- Upstream typically loads a GLTF or OBJ model that ships with an object-space normal map texture (e.g. `LeePerrySmith.glb` head model + `LeePerrySmith_obj.jpg` object-space normal map). Verify the current upstream model URL.
- `MeshNormalMaterial` is NOT what we want — we want a `MeshStandardMaterial` (or `MeshPhongMaterial`) with `normalMap` set AND `normalMapType: THREE.ObjectSpaceNormalMap` (the key distinguishing flag — `THREE.TangentSpaceNormalMap` is the default)
- Compare visually against tangent-space: optionally show two side-by-side meshes (left = object-space, right = tangent-space using the SAME RGB texture wrongly interpreted) so the difference is visible
- `DirectionalLight` + `AmbientLight` (or env from PMREM — match upstream)
- `OrbitControls` — rotating the camera should show how the normal-map detail responds to changing light angle
- If the GLTF/OBJ model is too heavy to fetch, an acceptable substitute is a `SphereGeometry(1, 32, 32)` with a procedurally-generated object-space normal map (compute `n = position / radius` per-fragment as the canonical sphere object-space normal, encode to RGB, write to a `DataTexture`) — document any substitution in SUMMARY
- Cleanup: dispose texture, geometry, material, controls, GUI, canvas
**Registry ID**: `webgl_materials_normalmap_object_space`
**Registry name**: `'Object-Space Normal Map'`
**Category**: `materials`
