### Task 1: Port WebGLMultipleScenesComparison

**File**: `demo/src/WebGLMultipleScenesComparison.tsx`
**Source**: `https://threejs.org/examples/#webgl_multiple_scenes_comparison`
**Reference scaffold**: `code/examples/webgl/misc/MultipleScenesComparison.tsx` (forbidden-pattern stub)
**Key features**:
- Two independent `THREE.Scene` instances rendered side-by-side in the SAME canvas via `renderer.setScissor(...)` + `renderer.setScissorTest(true)` — left half renders sceneA, right half renders sceneB
- Each scene contains the SAME geometry/camera but with DIFFERENT material or environment settings (upstream typically compares e.g. "with envmap" vs "without envmap", or two different tone-mapping operators) — verify exact comparison axis from upstream HTML and replicate
- A draggable HTML splitter (`<div>` overlay) on top of the canvas that the user drags horizontally; the X position drives the split point. On `pointermove` while dragging, update the splitter X and update the scissor regions on next render
- Both scenes share the same `PerspectiveCamera` + `OrbitControls` (or use synchronized cameras) so rotating one rotates both — visual goal: identical viewpoints, different rendering settings, splitter shows the diff
- Render order per frame: setScissor(0,0,splitX,h) → render sceneA → setScissor(splitX,0,w-splitX,h) → render sceneB
- Cleanup: dispose both scenes' geometries/materials/textures, controls, canvas, remove splitter div, remove pointer listeners
**Registry ID**: `webgl_multiple_scenes_comparison`
**Registry name**: `'Multiple Scenes Comparison (Split Viewport)'`
**Category**: `misc` (or whatever the existing `webgl_multiple_*` rows use — match neighbors in registry.ts)

### Task 2: Port WebGLSimpleGI

**File**: `demo/src/WebGLSimpleGI.tsx`
**Source**: `https://threejs.org/examples/#webgl_simple_gi`
**Reference scaffold**: `code/examples/webgl/misc/SimpleGI.tsx` (forbidden-pattern stub)
**Key features**:
- A "simple global illumination" technique — upstream uses a probe-sampling approach: for each lightprobe location (e.g. on a 4×4×4 grid in the scene), render a tiny cubemap of the surroundings, average the incident light, and use those samples to drive ambient/diffuse fill across the scene's materials
- Typical upstream pattern: a Cornell-box-style scene (colored walls, central object) where the "bounce" color of the walls visibly tints the central object via the simple-GI sampling
- Implementation: `THREE.CubeCamera` at each probe location → render the scene to a small `WebGLCubeRenderTarget` (e.g. 16×16 per face) → extract average colors → use those as `MeshLambertMaterial` ambient term or set `scene.environment` from the averaged probe
- This is a multi-pass render: probes update every N frames (not every frame) to keep cost manageable; main scene renders every frame using the most recent probe data
- `OrbitControls`
- Cleanup: dispose CubeCamera render targets, all scene geometries/materials/textures, controls, canvas
**Registry ID**: `webgl_simple_gi`
**Registry name**: `'Simple Global Illumination (Probe Sampling)'`
**Category**: `misc` (or match neighbors)

### Task 3: Port WebGLRendererPathtracer

**File**: `demo/src/WebGLRendererPathtracer.tsx`
**Source**: `https://threejs.org/examples/#webgl_renderer_pathtracer`
**Reference scaffold**: `code/examples/webgl/renderer/RendererPathtracer.tsx` (forbidden-pattern stub)
**Pre-flight package install (REQUIRED — confirmed missing)**:

3. Document the install (new package name + version) in this plan's SUMMARY.
**Key features**:
- Use `PathTracingRenderer` from `three-gpu-pathtracer` (NOT a replacement for `WebGLRenderer` — it RUNS ON TOP of WebGLRenderer and accumulates samples into a render target over multiple frames)
- Typical setup: load a GLTF model (upstream often uses the `DragonAttenuation.glb` or a simple cube assembly with transparent/glass materials so the pathtracer's caustics / soft-shadow / GI advantages are visible) → call `pathTracer.setSceneFromObject(model)` (or equivalent — check current `three-gpu-pathtracer` API on `https://github.com/gkjohnson/three-gpu-pathtracer`) → per frame call `pathTracer.update()` to accumulate one more sample → display the accumulated result via the standard renderer
- Sample counter overlay so the user sees the image progressively refining ("samples: 1 → 16 → 256" with visibly less noise each step)
- `OrbitControls` — note: orbiting RESETS the sample accumulation (pathtracer needs a static camera to accumulate cleanly); typical pattern hooks `controls.addEventListener('change', () => pathTracer.reset())`
- HDR environment map (PMREM-processed) so the pathtracer has rich incident light to sample
- Cleanup: dispose pathTracer (`pathTracer.dispose()` if API exists), scene geometries/materials/textures, controls, canvas
**Registry ID**: `webgl_renderer_pathtracer`
**Registry name**: `'GPU Pathtracer (three-gpu-pathtracer)'`
**Category**: `renderer` (match existing `webgl_renderer_*` rows)

### Task 4: Port WebGLGeometryTerrainRaycast

**File**: `demo/src/WebGLGeometryTerrainRaycast.tsx`
**Source**: `https://threejs.org/examples/#webgl_geometry_terrain_raycast`
**Reference scaffold**: `code/examples/webgl/geometry/GeometryTerrainRaycast.tsx` (forbidden-pattern stub)
**Key features**:
- A heightmap-displaced `PlaneGeometry` (e.g. 256×256 segments, 1024×1024 world units) — heightmap typically from `https://threejs.org/examples/textures/heightmap.png` or generated procedurally via Perlin noise (`ImprovedNoise` from `three/examples/jsm/math/ImprovedNoise.js`)
- Vertex Z (after rotating the plane -PI/2 X) displaced by sampling the heightmap → produces realistic terrain
- A `THREE.Raycaster` + `pointermove` → raycast against the terrain mesh → place a visible marker (e.g. a small cone or sphere) at the hit point so the user sees the cursor's projection onto the terrain surface
- Marker should follow the terrain surface smoothly as the cursor moves (this is the key feature — the marker "sticks to the surface" because the raycast hit is in world coords, not screen coords)
- Camera: angled-down `PerspectiveCamera` to show the terrain stretching to the horizon
- `OrbitControls` with `maxPolarAngle = Math.PI / 2.2` so the user can't fly under the terrain
- `Fog` for atmospheric depth (upstream uses `Fog`)
- Cleanup: dispose terrain geometry/material/texture, marker, controls, canvas, remove pointer listener
**Registry ID**: `webgl_geometry_terrain_raycast`
**Registry name**: `'Terrain + Raycast Picking'`
**Category**: `geometry` (match existing `webgl_geometry_terrain` row)

## Risk Notes (per-demo)

- **Task 1 (multi-scene comparison)**: Scissor-test bugs are common — forgetting `setScissorTest(true)` causes the whole canvas to render with the second scene over the first. Both scenes' resources double the memory footprint vs single-scene demos. Splitter drag listener must `stopPropagation` so it doesn't fight OrbitControls for pointer ownership.
- **Task 2 (simple GI)**: CubeCamera per-probe rendering is GPU-expensive — keep probe count modest (8–16 probes total, not 64+). Probes capture the scene WITHOUT themselves visible (they render to RT, not main canvas) → exclude them from main render via `cubeCamera.layers.set(...)` or similar. First N frames before probes converge will look unlit — that's why the screenshot wait is 5s+.
- **Task 3 (pathtracer)**: HIGHEST RISK task in this plan. `three-gpu-pathtracer` API has had breaking changes — verify the current API on the GitHub README before drafting (`https://github.com/gkjohnson/three-gpu-pathtracer`). The package requires `three-mesh-bvh` as a peer dep (also installed by Task 1 of Plan 21-01 — confirm before adding twice). Pathtracer has GPU memory requirements: scene must fit in a BVH; very large models may OOM. If the upstream model URL 404s, substitute with a smaller GLB and document. dv screenshot timing is critical — too early = noisy, too late = blocks downstream demos. Per-frame sample-counter overlay helps the developer judge convergence.
- **Task 4 (terrain raycast)**: heightmap.png URL on threejs.org is reliable. Vertex count from 256×256 = 65k vertices; on weak GPUs this can stutter — `BufferGeometry.setUsage(THREE.StaticDrawUsage)` (default) is correct since heightmap doesn't change. Raycast against 65k-tri mesh without BVH is fine for non-realtime cursor follow but might be choppy on dense terrain — if so, optionally apply `three-mesh-bvh` (installed by Task 1 of Plan 21-01) for acceleration; document the choice.
