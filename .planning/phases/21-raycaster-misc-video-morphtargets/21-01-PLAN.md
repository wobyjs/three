### Task 1: Port WebGLRaycasterBVH

**File**: `demo/src/WebGLRaycasterBVH.tsx`
**Source**: `https://threejs.org/examples/#webgl_raycaster_bvh` (fetch with WebFetch to get current asset URLs, model name, BVH options)
**Reference scaffold**: `code/examples/webgl/interactive/RaycasterBvh.tsx` (forbidden-pattern stub — DO NOT copy verbatim)
**Pre-flight package check**: `Grep "three-mesh-bvh" demo/package.json`. If absent, install: `cd D:\Developments\tslib\@woby\three\demo && pnpm add three-mesh-bvh`. Verify on npmjs.com/package/three-mesh-bvh first (legitimate, ~150k weekly DLs, maintained by gkjohnson, widely used in Three.js ecosystem). Document any install in SUMMARY.
**Key features**:
- Load a dense GLTF model (upstream typically `Horse.glb` or `XYZRGB-dragon.ply` or similar high-poly mesh; verify against upstream HTML — substitute with another threejs.org-hosted dense model if the referenced URL 404s, document substitution)
- Import `MeshBVH`, `acceleratedRaycast`, `computeBoundsTree`, `disposeBoundsTree` from `three-mesh-bvh`
- Monkey-patch the prototype: `THREE.BufferGeometry.prototype.computeBoundsTree = computeBoundsTree; THREE.BufferGeometry.prototype.disposeBoundsTree = disposeBoundsTree; THREE.Mesh.prototype.raycast = acceleratedRaycast`
- After loading: `mesh.geometry.computeBoundsTree()` builds the BVH
- `pointermove` listener computes normalized device coords `pointer.x = (event.clientX / w) * 2 - 1; pointer.y = -(event.clientY / h) * 2 + 1`
- `raycaster.setFromCamera(pointer, camera); const hits = raycaster.intersectObject(mesh)`
- Render a small visual hit marker at `hits[0].point` (e.g. a small red `SphereGeometry(0.02)` mesh whose position is set per-pointermove)
- Status overlay shows raycast time in ms (use `performance.now()` deltas) — the whole point of BVH is to make this microseconds-fast even on dense geometry
- `OrbitControls`, hemisphere + directional light, neutral background
- Cleanup: `mesh.geometry.disposeBoundsTree()`, dispose geometry/material/controls/marker/canvas, `removeEventListener('pointermove')`
**Registry ID**: `webgl_raycaster_bvh`
**Registry name**: `'BVH-Accelerated Raycaster (three-mesh-bvh)'`
**Category**: `raycaster` (or `interactive` — match the existing `webgl_raycaster*` rows' category in registry.ts; do NOT invent a new category)

### Task 2: Port WebGLRaycasterSprite

**File**: `demo/src/WebGLRaycasterSprite.tsx`
**Source**: `https://threejs.org/examples/#webgl_raycaster_sprite`
**Reference scaffold**: `code/examples/webgl/interactive/RaycasterSprite.tsx` (forbidden-pattern stub)
**Key features**:
- A scene populated with N (typically 50–200) `THREE.Sprite` instances — each sprite uses a `SpriteMaterial` with a small icon-style texture (upstream uses `sprite0.png`/`sprite1.png`/`sprite2.png` from `https://threejs.org/examples/textures/sprites/`; if those 404 you can generate a small `CanvasTexture` circle as substitute, document)
- Distribute sprites randomly in a cube volume (e.g. `position.set((Math.random()-0.5)*40, (Math.random()-0.5)*40, (Math.random()-0.5)*40)`) so the raycaster has many targets
- `pointermove` → `raycaster.setFromCamera(pointer, camera)` → `raycaster.intersectObjects(sprites)` (NOT `intersectObject` — array variant)
- On hit: highlight the hovered sprite (e.g. scale up by 1.5×, tint color to yellow) and revert previously-hovered sprite to default
- Status overlay: "Hovered: sprite #N" or "Hovered: none"
- `OrbitControls`, transparent or starry background
- Cleanup: dispose all sprite materials/textures, controls, canvas, remove pointer listener
**Registry ID**: `webgl_raycaster_sprite`
**Registry name**: `'Raycaster on Sprites (2D Billboards)'`
**Category**: same as Task 1 (`raycaster` or `interactive` — keep consistent)

### Task 3: Port WebGLRaycasterTexture

**File**: `demo/src/WebGLRaycasterTexture.tsx`
**Source**: `https://threejs.org/examples/#webgl_raycaster_texture`
**Reference scaffold**: `code/examples/webgl/interactive/RaycasterTexture.tsx` (forbidden-pattern stub)
**Key features**:
- A textured plane (or sphere) where the raycaster's UV-coordinate output drives a paint-into-texture effect
- Setup: load a base image (e.g. `https://threejs.org/examples/textures/uv_grid_opengl.jpg`) into a `CanvasTexture` whose backing canvas is mutable — i.e. create the texture from a `<canvas>` so you can draw into it per pointermove
- `pointermove` → raycast → `hits[0].uv` gives `(u, v)` in [0,1]² → convert to canvas pixel `(Math.floor(u * canvas.width), Math.floor((1 - v) * canvas.height))` (Y-flip!) → paint a small colored dot at that pixel via `ctx.fillStyle = ...; ctx.beginPath(); ctx.arc(px, py, 6, 0, Math.PI*2); ctx.fill()` → set `texture.needsUpdate = true`
- Visual result: as the user moves the mouse across the textured surface, a trail of colored dots appears on the texture in real time, perfectly tracking the cursor regardless of perspective angle
- `MeshBasicMaterial({ map: canvasTexture })`
- `OrbitControls`
- Cleanup: dispose texture, geometry, material, controls, canvas, remove pointer listener
**Registry ID**: `webgl_raycaster_texture`
**Registry name**: `'Raycaster → UV → Paint into Texture'`
**Category**: same as Tasks 1–2

### Task 4: Port WebGLInteractiveBufferGeometry

**File**: `demo/src/WebGLInteractiveBufferGeometry.tsx`
**Source**: `https://threejs.org/examples/#webgl_interactive_buffergeometry`
**Reference scaffold**: `code/examples/webgl/interactive/InteractiveBufferGeometry.tsx` (forbidden-pattern stub)
**Key features**:
- A large `BufferGeometry` whose `position` and `color` attributes are dynamic — upstream typically renders a high-vertex-count triangulated mesh (e.g. a 100×100 grid of triangles, ~20k vertices) where vertex colors animate based on cursor position
- Per pointermove: raycast against an invisible plane (or directly use NDC coords) → for each vertex within radius R of the hit point, recolor toward a highlight hue and slightly displace position (e.g. `position.array[i*3+2] += amount` to bump Z) → call `position.needsUpdate = true` and `color.needsUpdate = true`
- Visual goal: cursor leaves a glowing "ripple" of color + height across the dense triangle mesh
- Use `attribute.setUsage(THREE.DynamicDrawUsage)` on position and color so the GPU re-uploads efficiently
- `MeshBasicMaterial({ vertexColors: true, side: THREE.DoubleSide })` (or MeshStandardMaterial with vertexColors + lighting — match upstream)
- `OrbitControls`
- Cleanup: dispose geometry, material, controls, canvas, remove pointer listener
**Registry ID**: `webgl_interactive_buffergeometry`
**Registry name**: `'Interactive BufferGeometry (Vertex Painting)'`
**Category**: same as Tasks 1–3

## Risk Notes (per-demo)

- **Task 1 (BVH)**: `three-mesh-bvh` package likely needs install — verify via grep first. Dense GLTF model may be slow to fetch on a cold cache (wait up to 5s before screenshot). Prototype monkey-patch must run BEFORE first raycast call or BVH is ignored.
- **Task 2 (sprite)**: sprite texture URLs (`sprite0.png` etc.) on threejs.org are reliable but if they 404 the entire scene goes blank — fall back to a procedurally-generated `CanvasTexture` (small circle) and document. Sprite-hover highlight may not be visible in a static screenshot without simulated mouse input — that's acceptable.
- **Task 3 (texture)**: `uv_grid_opengl.jpg` from threejs.org is stable. Watch the UV-flip carefully: `(u, v)` is in [0,1] with v=0 at bottom; canvas pixel y=0 is at top, so painting requires `y = (1 - v) * canvas.height`. Common bug.
- **Task 4 (interactive buffergeometry)**: no external assets — lowest-risk. Vertex count should be 5k–20k (sweet spot for visible ripple without GPU stall). Per-pointermove loop should be O(N) not O(N²) — pre-compute squared-distance once per pointer event.
