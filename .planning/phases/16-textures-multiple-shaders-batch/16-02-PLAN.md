# Phase 16 Plan 02: Multiple/Shadowmap/Raycast Batch

## Context

This is the middle plan of Phase 16 (3 plans total). 16-01 covers MarchingCubes rewrite + 5 texture demos (in progress). 16-02 (this plan) groups 6 demos that share a "multi-pass / multi-target / multi-view" theme: split-screen multi-camera rendering, multi-render-target deferred shading, advanced shadowmap techniques, and sprite raycasting. These were grouped because they all exercise either multiple render passes or non-trivial WebGLRenderer state changes per frame — keeping them in one plan keeps the executor in the same mental model. All 6 IDs verified absent from `demo/src/registry.ts` at planning time.

## Tasks

### Task 1: Port WebGLMultipleViews

**File**: `demo/src/WebGLMultipleViews.tsx`
**Source**: threejs.org/examples/#webgl_multiple_views
**Key features**:
- Single `WebGLRenderer` with one canvas
- 4 `PerspectiveCamera` instances configured as separate views (top-left, top-right, bottom-left, bottom-right)
- Per-frame loop sets `renderer.setScissorTest(true)`, calls `renderer.setViewport(x, y, w, h)` and `renderer.setScissor(x, y, w, h)` per view, then `renderer.render(scene, camera)` 4 times
- Shared scene with a few `Mesh` objects (e.g., spinning torus knot, ground plane) and at least one `DirectionalLight`
- View metadata array `{ left, bottom, width, height, background, eye, up, fov, updateCamera }` driving the loop
- Optional `OrbitControls` on the main (front) view only
**Registry ID**: `webgl_multiple_views`
**Category**: `cameras`

### Task 2: Port WebGLMultipleElementsText

**File**: `demo/src/WebGLMultipleElementsText.tsx`
**Source**: threejs.org/examples/#webgl_multiple_elements_text
**Key features**:
- Single `WebGLRenderer` rendering full-window, but z-index behind HTML content
- DOM layout: paragraphs of placeholder `<p>` text with multiple inline `<div class="scene">` placeholders scattered between them
- Per-frame: for each scene-div, read its `getBoundingClientRect()`, set scissor + viewport to that rect (skip if off-screen), render that scene/camera into the rect
- Each scene contains a different geometry (e.g., Box, Sphere, Torus, Dodecahedron) lit by ambient + directional
- Page scrolls; viewports follow scroll position
- Use `requestAnimationFrame` driven by woby effect, cancellable via `_cleanupFn`
**Registry ID**: `webgl_multiple_elements_text`
**Category**: `cameras`

### Task 3: Port WebGLMultipleRendertargets

**File**: `demo/src/WebGLMultipleRendertargets.tsx`
**Source**: threejs.org/examples/#webgl_multiple_rendertargets
**Key features**:
- `WebGLRenderTarget(width, height, { count: 2 })` to produce 2 color attachments (diffuse + normal) — requires WebGL2
- `RawShaderMaterial` with GLSL 3.00 ES shaders writing `pc_fragData[0]` (diffuse) and `pc_fragData[1]` (normal) in the fragment shader
- Two-pass render: first pass renders scene (a textured `BoxGeometry`) into the MRT; second pass uses a fullscreen quad with another `RawShaderMaterial` that samples both targets to produce a split-screen composite
- `renderTarget.textures[0]` and `[1]` are bound as `tDiffuse` and `tNormal` uniforms on the composite material
- Resize handler updates both target dimensions
**Registry ID**: `webgl_multiple_rendertargets`
**Category**: `advanced`

### Task 4: Port WebGLShadowmapPCSS

**File**: `demo/src/WebGLShadowmapPCSS.tsx`
**Source**: threejs.org/examples/#webgl_shadowmap_pcss
**Key features**:
- `DirectionalLight` with `castShadow = true`, `shadow.mapSize.set(1024,1024)`, `shadow.camera` framed around scene
- Monkey-patch `THREE.ShaderChunk.shadowmap_pars_fragment` to inject PCSS sampling code (blocker search → penumbra estimate → PCF kernel with variable radius)
- Scene: ground plane, several `Mesh` (sphere + box stack) all casting/receiving shadows
- `PCFSoftShadowMap` enabled on renderer
- OrbitControls
- The patched chunk must be restored on cleanup (`THREE.ShaderChunk.shadowmap_pars_fragment = originalChunk`) so other demos aren't affected
**Registry ID**: `webgl_shadowmap_pcss`
**Category**: `shadows`

### Task 5: Port WebGLShadowmapProgressive

**File**: `demo/src/WebGLShadowmapProgressive.tsx`
**Source**: threejs.org/examples/#webgl_shadowmap_progressive
**Key features**:
- Import `ProgressiveLightMap` from `three/examples/jsm/misc/ProgressiveLightMap.js`
- Scene with static receiver mesh (ground + walls) and moving caster meshes (e.g., bouncing spheres)
- `progressiveSurfaceMap = new ProgressiveLightMap(renderer, lightmapResolution=1024)`
- `progressiveSurfaceMap.addObjectsToLightMap(receivers)` then per-frame `progressiveSurfaceMap.update(camera, blendWindow=blendWindowVal, blurEdges=true)`
- `DirectionalLight` whose position is randomly jittered each frame to integrate soft shadows over time
- GUI: blend window, blur, light-map resolution toggle
- OrbitControls
**Registry ID**: `webgl_shadowmap_progressive`
**Category**: `shadows`

### Task 6: Port WebGLRaycastSprite

**File**: `demo/src/WebGLRaycastSprite.tsx`
**Source**: threejs.org/examples/#webgl_raycast_sprite
**Key features**:
- Many `Sprite` instances (50–200) placed at random positions; each uses a `SpriteMaterial` with a small `CanvasTexture` (or shared texture)
- `Raycaster` with `params.Sprite = {}` (default uses bounding-box hit test, but verify against three.js source — sprite intersection uses transformed quad)
- `pointermove` handler updates a `Vector2` mouse coord; raycaster against the sprite array; intersected sprite changes color (e.g., `sprite.material.color.set(0xff0000)`) and prior intersect resets
- Camera with `OrbitControls`; sprites face camera automatically (sprite behavior)
- Cleanup removes the `pointermove` listener and disposes the canvas texture
**Registry ID**: `webgl_raycast_sprite`
**Category**: `interactive`
